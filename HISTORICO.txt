PROJETO: Manual do Montador — Ferragens e Acessorios (Blum, Hafele, Hettich)

===========================================================================
O QUE E O PROJETO
===========================================================================
App de apoio para montadores de moveis: catalogo de acessorios/ferragens
organizados por fabricante (Blum, Hafele, Hettich), com imagem,
especificacoes e passo a passo de instalacao. A base e um PROTOTIPO em
HTML/CSS/JS puro, empacotado com Capacitor num app Android instalavel
(APK). Desde 2026-07-12 tem um BACKEND de verdade (Node/Express +
Postgres) - o catalogo e o login sao compartilhados entre todo mundo que
usa o app; o app continua funcionando OFFLINE pra CONSULTAR (usa uma
copia em cache), so precisa de internet pra login e pra adicionar/
editar/remover.

Estrutura principal:
- ESCOPO.md            -> documento de escopo (funcionalidades, telas, fluxo)
- HISTORICO.txt         -> este arquivo (log de tudo que foi feito)
- prototipo/            -> codigo-fonte web (HTML/CSS/JS), fonte de verdade
    index.html          -> lista de fabricantes (checa atualizacao aqui)
    fabricante.html     -> lista de acessorios de um fabricante (ex-marca.html)
    acessorio.html      -> detalhe do acessorio
    passo.html          -> passo a passo de instalacao (imagem + texto)
    adicionar.html      -> cadastro/edicao de fabricante e acessorio proprio (upload de fotos, exige login com permissao)
    login.html          -> tela de login / conta (usuario logado, sair)
    favoritos.html      -> lista de acessorios marcados como favoritos
    css/style.css        -> estilo visual (tela em formato de celular)
    js/api.js              -> fala com o backend (fetch com token, timeout, erro de "sem internet")
    js/data.js            -> cache local do catalogo (sincronizado do backend) + favoritos (so local)
    js/app.js             -> funcoes auxiliares (imagens, logos, query params, botao de favorito)
    js/auth.js             -> login de verdade via backend (token JWT guardado no localStorage)
    js/cropper.js          -> editor de recorte/zoom de foto antes de salvar
    js/menu.js             -> menu lateral (favoritos + versao do app)
    js/versao.js            -> numero da versao atual do app (VERSAO_APP)
    js/atualizacao.js       -> verifica versao nova no GitHub Releases e avisa
    assets/logos/          -> logos dos 3 fabricantes originais (catalogo)
    assets/produtos/       -> fotos dos acessorios/passos do catalogo original
    assets/README.md       -> lista exata dos arquivos de imagem esperados
- backend/              -> API (Node/Express + Postgres) - ver ESCOPO.md > "Backend"
    server.js            -> rotas (auth, catalogo, fabricantes, acessorios)
    db.js / schema.sql    -> conexao com Postgres e schema das tabelas
    auth.js               -> hash de senha, gera/valida token JWT
    seed.js               -> popula o catalogo original + usuario de exemplo
- android/              -> projeto nativo Android gerado pelo Capacitor
                           (gerado automaticamente, nao editar na mao;
                           refletir mudancas rodando "npx cap sync android")
- capacitor.config.json -> config do Capacitor (appId, nome, pasta web)
- package.json           -> dependencias do Capacitor (npm)
- .gitignore             -> ignora node_modules/, build do Android, *.apk

- Os builds Android (assembleDebug/assembleRelease) vinham falhando
  direto com erro "Unable to delete directory" em pastas dentro de
  android/*/build. Causa: o projeto ficava dentro de uma pasta
  sincronizada pelo OneDrive, que trava arquivos brevemente enquanto
  sincroniza - e o Gradle cria/apaga MUITOS arquivos pequenos durante o
  build.
- A pedido do usuario, o projeto inteiro foi movido de
  "OneDrive\Documentos\Desktop\PASTA_PESSOAL\TESTEAPP" pra

  keystore de assinatura, node_modules, tudo preservado - só o caminho no
  disco mudou. O historico do git (commits, tags, remoto configurado)
  continua intacto.
- Mesmo fora do OneDrive, o mesmo tipo de erro ainda acontecia (mais
  raro) - identificadas mais duas causas, ambas sem relacao com OneDrive:
    1. O recurso de "observador de arquivos" (file-system watch) do
       proprio Gradle, que no Windows pode manter um arquivo aberto por
       baixo dos panos e entrar em conflito com o proprio Gradle tentando
       apagar essa mesma pasta. Corrigido adicionando
       "org.gradle.vfs.watch=false" em android/gradle.properties.
    2. O "daemon" do Gradle (processo que fica rodando em segundo plano
       entre builds pra ser mais rapido) as vezes mantinha handles de
       arquivo presos entre uma tentativa e outra. Resolvido rodando o
       build com a flag "--no-daemon" (cada build roda num processo
       novo, que fecha certinho no final).
- Com as duas correcoes, o build de release passou a funcionar de forma
  confiavel. Gerado e assinado o APK da v1.0.1 (mesma assinatura/keystore
  da v1.0.0 - instala como atualizacao normal, sem precisar desinstalar):

- Se builds futuros voltarem a falhar com "Unable to delete directory",
  o jeito de resolver e: parar o daemon (gradlew --stop), apagar a pasta
